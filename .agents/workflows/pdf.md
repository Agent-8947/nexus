Этот ворклоу активирует протокол "Nexus Prime DOC" для создания документов как глубокой компиляции всего контекста проекта.

### 1. Когнитивная подготовка (Смысловое Проектирование)

Прежде чем писать код — активируем роли **Архитектора Смыслов** и **Архитектора Документа**.

- **Цель**: Превратить сухие данные в стратегический цифровой актив.
- **Действие**: Читаем Knowledge Items (KI) и историю для выявления "глубинных нервов" проекта.
- **Check KIs**: Анализ паттернов мышления пользователя, чтобы документ "говорил" на его языке и в его системе ценностей.

### 2. Стратегическое рассуждение (Архитектура Смыслов)

Это этап проектирования "ДНК документа". Архитектор Смыслов выстраивает логическую цепь:

- **Априорная цель**: Что пользователь просит буквально?
- **Скрытая цель (Intent)**: Какую проблему документ решает стратегически? (Напр. "узаконить хаос" или "спроектировать будущее").
- **Семантическая карта**: Определение 3–5 ключевых смыслов, которые должны пройти красной нитью через весь PDF.
- **Вероятностный стек**: React-PDF v3 для реализации Visual Logic, где форма — это продолжение смысла.

### 3. Реконструкция контекста (Context Reconstruction)

Сбор "цифрового следа" из всей истории диалога:

- **Архитектурные ADR**: Какие решения (Rust, Python, Go) уже утверждены?
- **Эволюция идей**: Что вначале было хаосом, а теперь стало структурой?
- **Терминология**: Использование специфического сленга пользователя для создания ощущения "своего" продукта.

### 4. Состязательный самоанализ (Архитектор Документа)

Проверка документа на прочность и честность перед сборкой:

- **Информационная плотность**: Не "пустой" ли это документ?
- **Логическая чистота**: Нет ли смысловых противоречий с ранее принятыми ADR?
- **Эстетическая верность**: Работает ли Visual Logic (Visual Hierarchy) на усиление ключевых смыслов?
*Если находим "воду" или слабые аргументы — Архитектор Смыслов перестраивает структуру.*

### 5. Проектирование скелета (Map Before Code)

Цепочка: **ЦЕЛЬ → СУЩНОСТИ → СТРУКТУРА → КОД → ФАЙЛ**.
Создаем PROJECT_MAP.md для документа, если он больше 10 страниц.

### 6. Программная сборка (Multi-Format Assembly)

Документ собирается как ПО. Текст и оформление — это данные, которые подаются на вход функциям генерации:

- **DOCX**: Для редактируемых отчетов с полной версткой.
- **PDF**: Luxury V3 Edition на базе `@react-pdf/renderer` для безупречного рендеринга.

### 8. Финализация

- Проверка визуальной целостности (Premium Check) для всех трёх форматов (HTML, DOCX, PDF).
- Копирование в `PROJECT/outputs/`.
- Представление результата с кратким обоснованием принятых решений.
